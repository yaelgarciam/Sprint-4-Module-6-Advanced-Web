SECCIÓN 0 — PANORAMA GENERAL DE LA APLICACIÓN
═══════════════════════════════════════

Escribe una introducción completa de 1–2 páginas que cubra:

0.1 ¿Qué es TutorBot Campus?
- Es una plataforma de tutoría adaptativa universitaria construida con arquitectura de microservicios.
- Su propósito es evaluar automáticamente las respuestas de los estudiantes, detectar lagunas en su aprendizaje (learning gaps) y eventualmente sugerir rutas de estudio personalizadas.
- Está diseñada como un sistema multi-agente donde cada servicio tiene una responsabilidad específica.

0.2 Los dos microservicios que trabajaremos
Genera una tabla con esta información:

| Servicio              | Puerto | Responsabilidad                                                                 |
|-----------------------|--------|---------------------------------------------------------------------------------|
| evaluator-service     | 8082   | Recibe las respuestas de los estudiantes, las evalúa (con IA o por reglas), asigna puntaje y genera retroalimentación. Persiste cada evaluación en Oracle. |
| gap-detector-service  | 8084   | Analiza las evaluaciones, detecta conceptos donde el estudiante tiene debilidades (gaps), calcula su severidad y confianza. Persiste los gaps en Oracle. |

0.3 ¿Por qué base de datos Oracle?
- En la industria enterprise, Oracle es una de las bases de datos más utilizadas.
- Aprenderán a usar JPA/Hibernate con un motor relacional real, manejar secuencias (SEQUENCE), tipos CLOB, y escribir scripts DDL idempotentes.
- Hibernate está configurado con `ddl-auto=validate`: NO crea tablas, las VALIDA. Ustedes deben crear el schema antes de correr los servicios.

0.4 ¿Cómo se conectan los servicios entre sí?
- En esta primera parte del laboratorio, cada servicio funciona de forma INDEPENDIENTE: entrada y salida de datos via REST (Postman) ↔ Oracle.
- NO necesitan RabbitMQ para empezar. Los servicios tienen un feature flag (`tutorbot.messaging.enabled`) que deshabilita la mensajería.
- Diagrama conceptual (textual, que el alumno visualice):

  [Postman] → REST → [evaluator-service :8082] → Oracle (tabla EVALUATIONS)
  [Postman] → REST → [gap-detector-service :8084] → Oracle (tablas GAPS, COURSES, TOPICS, SUBTOPICS)

0.5 ¿Para qué usaremos RabbitMQ? (Parte 2)
- RabbitMQ es un message broker que permite comunicación asíncrona entre servicios.
- Cuando la mensajería se habilita, el flujo se convierte en:
  1. Un evento "student.answered" llega a evaluator-service via RabbitMQ
  2. evaluator-service evalúa → persiste en Oracle → publica "evaluation.completed"
  3. gap-detector-service escucha "evaluation.completed" → analiza gaps → persiste → publica "gap.detected"
- Esto es el patrón Event-Driven Architecture, fundamental en sistemas distribuidos.

0.6 ¿Qué vendrá después?
- Integración con Ollama (LLM local) para evaluación con IA
- Service Discovery con Eureka
- API Gateway
- Circuit Breaker con Resilience4j
- Frontend React

Incluye un diagrama de arquitectura de alto nivel (textual o mermaid) que muestre los microservicios, Oracle, RabbitMQ, y las flechas de comunicación.

3.1 ¿Qué es RabbitMQ y por qué lo necesitamos?
- RabbitMQ es un message broker open-source que implementa el protocolo AMQP.
- Permite que los servicios se comuniquen de forma asíncrona: un servicio publica un mensaje en una cola, otro lo consume.
- Ventajas: desacoplamiento, resilencia (si un servicio se cae, los mensajes esperan en la cola), escalabilidad.

3.4 Habilitar la mensajería en el código
Explica que el código ya tiene toda la infraestructura de RabbitMQ implementada pero DESHABILITADA mediante un feature flag. Esto es un patrón común en la industria llamado "Feature Flags" o "Feature Toggles".

Editar `application-dev.properties` en AMBOS servicios. Los archivos están en:
```
backend/evaluator-service/src/main/resources/application-dev.properties
backend/gap-detector-service/src/main/resources/application-dev.properties
```

Cambiar estas líneas:
```properties
# ANTES (messaging deshabilitado):
tutorbot.messaging.enabled=false
spring.autoconfigure.exclude=org.springframework.boot.autoconfigure.amqp.RabbitAutoConfiguration
# spring.rabbitmq.host=localhost
# spring.rabbitmq.port=5672

# DESPUÉS (messaging habilitado):
tutorbot.messaging.enabled=true
# spring.autoconfigure.exclude=org.springframework.boot.autoconfigure.amqp.RabbitAutoConfiguration
spring.rabbitmq.host=localhost
spring.rabbitmq.port=5672
```

Resumen de los 3 cambios en cada archivo:
1. `tutorbot.messaging.enabled` → cambiar de `false` a `true`
2. `spring.autoconfigure.exclude` → COMENTAR la línea (poner # al inicio)
3. `spring.rabbitmq.host` y `spring.rabbitmq.port` → DESCOMENTAR (quitar el #)

Explica qué hace cada cambio técnicamente:
- `tutorbot.messaging.enabled=true` activa los beans anotados con `@ConditionalOnProperty`: las clases `EvaluatorInfrastructureConfig`, `EvaluatorEventListener`, `GapDetectorMessagingConfig` y `GapDetectorEventListener`.
- Quitar la exclusión de `RabbitAutoConfiguration` permite que Spring Boot cree automáticamente la conexión a RabbitMQ.
- Las propiedades `spring.rabbitmq.*` le dicen a Spring dónde está RabbitMQ.

SECCIÓN 5 — GLOSARIO
═══════════════════════════════════════

| Término | Definición |
|---------|------------|
| Microservicio | Servicio independiente con una responsabilidad específica |
| JPA | Java Persistence API — estándar de mapeo objeto-relacional |
| Hibernate | Implementación de JPA que genera SQL automáticamente |
| ddl-auto=validate | Hibernate solo verifica que las tablas existan, no las modifica |
| SEQUENCE | Generador de IDs numéricos en Oracle (diferente a AUTO_INCREMENT de MySQL) |
| CLOB | Character Large Object — columna para textos largos en Oracle |
| Feature Flag | Bandera en configuración que habilita/deshabilita funcionalidad sin cambiar código |
